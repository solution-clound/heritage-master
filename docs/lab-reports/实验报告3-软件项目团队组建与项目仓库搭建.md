### 3.5 团队协作基础规则

#### （1）Git 仓库使用规范

| 规则项 | 具体要求 |
|--------|----------|
| **分支管理** | `main` 为稳定分支，禁止直接推送；所有开发在功能分支上进行 |
| **提交信息格式** | 采用 Conventional Commits 规范：`<type>: <description>` |
| **提交类型** | `feat`（新功能）、`fix`（修复）、`docs`（文档）、`test`（测试）、`refactor`（重构） |
| **推送频率** | 每完成一个独立功能模块即推送，避免大量代码堆积 |
| **代码合并** | 功能分支合并到 `main` 前，需至少1人进行Code Review |

#### （2）代码提交要求

- 每次提交必须有清晰的提交信息，说明修改内容
- 禁止提交包含敏感信息的文件（如 `.env`、API密钥等）
- 提交前检查代码是否能正常运行，避免提交错误代码
- 同一功能的修改尽量合并为一次提交，保持提交历史整洁

#### （3）日常沟通方式

| 沟通场景 | 工具/方式 | 频率 |
|----------|----------|------|
| 日常交流 | 微信群聊 | 实时 |

| 进度同步 | 每周例会（线上/线下） | 每周1次 |
| 代码协作 | GitHub Pull Request | 按需 |
| 文档共享 | 腾讯文档/飞书 | 按需 |

#### （4）开发环境统一

- **Python版本**：3.12+

## 四、主要问题和解决方案

### 问题1：团队成员Git操作不熟练
- **问题描述**：部分团队成员此前未使用过Git进行多人协作，不清楚分支管理与合并流程
- **解决方案**：项目组长组织了一次Git基础操作培训，涵盖clone、branch、add、commit、push、pull、merge等常用命令，并在实际项目中边做边学

### 问题2：多人同时修改同一文件导致冲突
- **问题描述**：在开发初期，前后端同学同时修改了部分配置文件，产生了合并冲突
- **解决方案**：制定明确的文件分工规则——后端同学负责 `src/` 和 `web/app.py`，前端同学负责 `web/frontend/`，测试同学负责 `tests/`。同时约定每日下班前同步一次代码（`git pull`）

### 问题3：敏感信息泄露风险
- **问题描述**：项目中包含API密钥等敏感配置（如DeepSeek API Key），需要避免提交到公开仓库
- **解决方案**：在 `.gitignore` 中添加 `.env` 规则，同时创建 `.env.example` 文件记录必要的环境变量名（不含真实密钥），新成员根据示例文件自行配置

### 问题4：提交信息不规范
- **问题描述**：初期团队成员的提交信息比较随意（如"完成"、"更新"等），缺乏可读性
- **解决方案**：制定统一的提交信息规范，采用 Conventional Commits 格式，并在团队群中进行规范宣导

## 六、心得体会与自我评价

### 心得体会

通过本次实验，团队完成了以下方面的学习和实践：

**1. 团队组建方面**

本次实验以4人小组的形式完成团队组建，每位成员根据自身技术特长分配了相应的角色。在分工过程中，我们深刻体会到合理分工的重要性——将后端开发、前端开发、数据处理、测试文档分别分配给最擅长对应领域的同学，能够最大限度发挥每位成员的优势，提高整体开发效率。

**2. Git版本控制方面**

通过实际操作，团队成员掌握了Git的基本工作流程：clone -> branch -> add -> commit -> push -> merge。特别是分支管理策略的学习，让我们理解了为什么需要在独立分支上开发再合并到主分支，以及如何通过规范的提交信息保持清晰的版本历史。在实际协作中也遇到了文件冲突的问题，通过及时沟通和明确分工规则得到了解决。

**3. 项目仓库管理方面**

学习了如何配置 .gitignore 保护敏感信息，如何通过 .env.example 提供配置模板，以及如何组织清晰的项目目录结构。一个规范的项目仓库不仅方便团队协作，也是项目专业性的体现。

**4. 协作规范方面**

制定了提交信息规范、分支管理策略、日常沟通机制等团队协作规则。这些规范看似简单，但在实际开发中对保持团队高效运转起到了重要作用。
